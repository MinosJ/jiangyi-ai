整个项目是经典的前后端分离架构：前端负责 UI 渲染和用户交互，后端负责 AI 推理和数据检索。两者通过 HTTP API 通信，部署在不同平台上。

02前端架构（你熟悉的部分）

技术栈

页面结构

/ 首页（Server Component）

静态展示页，SSG 预渲染。包含 Hero、Skills、Projects、Timeline、CTA 五个组件区块。纯展示，不请求后端。

/chat 对话页（Client Component）

交互式聊天界面。标记为 "use client"，使用 useState 管理消息列表，通过 SSE 流实时接收 AI 回复。

聊天的数据流（前端视角）

用户输入问题 → fetch POST /api/chat → ReadableStream 读取 → 逐 token 追加到 state → React 重新渲染

关键代码：流式聊天（api.ts）

// 用 fetch + ReadableStream 实现 SSE 消费
const reader = response.body?.getReader();
const decoder = new TextDecoder();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;

  buffer += decoder.decode(value, { stream: true });
  // 按 "data: {...}\n\n" 格式解析每个事件
  for (const line of lines) {
    if (line.startsWith("data: ")) {
      const parsed = JSON.parse(line.slice(6));
      yield parsed; // { type: "token", data: "你" }
    }
  }
}

这就是前端实现"打字机效果"的核心：不是 WebSocket，而是 SSE（Server-Sent Events）。后端每生成一个 token 就推一条数据，前端用 AsyncGenerator 逐条消费，追加到 React state 触发重新渲染。

类比前端概念：SSE 就像你用 EventSource 监听服务器推送，只不过这里手动用 ReadableStream 实现了更灵活的解析。跟 WebSocket 的区别是 SSE 是单向的（服务器→客户端），用普通 HTTP 就行。

Markdown 渲染

AI 返回的内容是纯 Markdown 文本。前端用 react-markdown + remark-gfm（GitHub 风格 Markdown）+ rehype-highlight（代码高亮）渲染成 HTML。通过自定义 components 映射（h1/h2/code/table 等），给每个 HTML 元素加上 Tailwind 样式类。

核心流程：一次聊天请求的完整旅程

Step 1：路由接收请求（routers/chat.py）

前端 POST { messages: [...], stream: true } 到 /api/chat。FastAPI 用 Pydantic 模型自动校验请求格式（就像 Zod 验证前端表单一样）。

Step 2：RAG 检索（services/rag.py）

取出用户最新一条消息，去 ChromaDB 向量数据库里做语义搜索：

  1. 把问题文本通过 BGE 模型转成一个 512 维的数字向量（Embedding）
  2. 在 ChromaDB 中找到余弦相似度最高的 5 段文档片段
  3. 这些片段可能来自你的简历、项目 README、面试题等

类比：想象一个超级搜索引擎，不是按关键词匹配，而是按"语义"找最相关的内容。问"你做过什么项目"会找到项目经历，而不是仅仅包含"项目"二字的段落。

Step 3：构建 Prompt（services/llm.py）

把检索到的文档片段拼接成"参考资料"，加上预设的系统提示词（System Prompt），一起发给 DeepSeek：

messages = [
  {
    "role": "system",
    "content": "你是蒋毅的 AI 数字分身..." + 检索到的参考资料
  },
  { "role": "user", "content": "你做过哪些项目？" }
]

System Prompt 里定义了 AI 的人设（31岁、大前端工程师、扇贝技术主管等）和回答规则（第一人称、不编造、用 Markdown 格式等）。

Step 4：流式生成（DeepSeek API → SSE）

调用 DeepSeek 的 Chat Completion API（跟 OpenAI API 格式完全一样），设置 stream=True。DeepSeek 边思考边一个 token 一个 token 地返回，后端每收到一个就包装成 SSE 事件推给前端：

async for chunk in stream:
    token = chunk.choices[0].delta.content
    yield f"data: {json.dumps({'type': 'token', 'data': token})}\n\n"

04RAG 是什么？（核心 AI 概念）

RAG = Retrieval-Augmented Generation（检索增强生成）

没有 RAG: 用户: "你做过什么项目？" → 直接问 DeepSeek → DeepSeek 瞎编（它不知道蒋毅是谁） 有 RAG: 用户: "你做过什么项目？" → 先从知识库检索相关文档片段 → 把文档片段 + 问题一起发给 DeepSeek → DeepSeek 基于真实资料回答（不编造）

简单说：RAG 就是给 AI 开卷考试。AI 本身不知道你的信息，但你把相关资料"塞"给它，它就能基于这些资料回答。

知识库是怎么建的？（数据导入流程）

简历/README.md → 读取文件内容 → 切成 500 字的片段 → BGE 模型转成向量 → 存入 ChromaDB

为什么要切片？

大模型有上下文长度限制（DeepSeek 约 64K tokens），不能把所有文档一股脑塞进去。所以先切成小片段，每次只检索最相关的 5 段，既省 token 又精准。

为什么要转成向量？

向量是一串数字（如 [0.12, -0.34, 0.56, ...]），可以用数学方法计算两段文本的"语义距离"。比如"React 前端开发"和"Web 前端工程师"的向量会很接近，但和"厨师招聘"的向量很远。

05关键技术组件详解

ChromaDB 向量数据库

类似 SQLite 之于关系型数据库——轻量级、嵌入式、文件存储。存的不是表格行列，而是向量+原文+元数据。查询方式不是 SQL，而是"给我一个向量，找最近的 N 个"。

数据存在 ./chroma_db/ 文件夹里，不需要单独启动数据库服务。

BGE-small-zh Embedding 模型

北京智源研究院（BAAI）出的中文向量模型，专门把中文文本转成 512 维向量。"small" 版本只有 ~100MB，可以在 CPU 上运行，不需要 GPU。

这个模型在后端本地运行，不调用任何 API，所以免费且无延迟。

DeepSeek API 大语言模型

国产大模型，API 接口兼容 OpenAI 格式，所以代码里直接用 OpenAI 的 Python SDK（AsyncOpenAI），只是把 base_url 改成了 DeepSeek 的地址。

价格很便宜（约 ¥1/百万 token），个人项目几乎不花钱。

LangChain AI 工具链

本项目只用了它的 RecursiveCharacterTextSplitter——一个智能文本分割器，会按段落→句子→字符的优先级切片，保证每片语义完整。

类比前端：就像 lodash.chunk()，但更聪明。

06本地开发怎么跑

前端（你会的）

cd frontend
npm install
npm run dev
# → http://localhost:3000

标准 Next.js 开发流程，Tailwind 热更新，没什么特别的。

后端（新东西）

cd backend
python -m venv .venv        # 创建虚拟环境（≈ 独立的 node_modules）
source .venv/bin/activate   # 激活虚拟环境
pip install -r requirements.txt  # 安装依赖（≈ npm install）
uvicorn app.main:app --reload    # 启动开发服务器
# → http://localhost:8000

--reload 相当于 nodemon，改代码自动重启。

后端首次启动会自动执行数据导入（ingest），加载你的简历和项目文档到 ChromaDB。之后重启不会重复导入，除非手动调用 POST /api/ingest。

07线上部署架构

┌────────────────────┐ ┌─────────────────────┐ │ Vercel │ HTTP │ Belmo │ │ (前端) │ ──────→ │ (后端) │ │ │ │ │ │ Next.js SSG/SSR │ │ FastAPI + Python │ │ 全球 CDN 分发 │ │ Docker 容器运行 │ │ 自动 HTTPS │ │ 自动 HTTPS │ │ │ │ │ │ 环境变量: │ │ 环境变量: │ │ NEXT_PUBLIC_API_ │ │ DEEPSEEK_API_KEY │ │ URL=belmo域名 │ │ FRONTEND_URL= │ │ │ │ vercel域名 │ └────────────────────┘ └─────────────────────┘ ▲ │ │ ▼ │ ┌──────────┐ GitHub 仓库 │ DeepSeek │ MinosJ/jiangyi-ai │ 云端 API │ 推送后自动部署 └──────────┘

前端部署（Vercel）

跟你部署普通 Next.js 项目一样：连接 GitHub 仓库 → Vercel 自动检测 Next.js → 每次 push 自动构建和部署。唯一的区别是需要设置 NEXT_PUBLIC_API_URL 环境变量指向后端地址。

后端部署（Belmo）

Belmo 是一个 PaaS 平台（类似 Heroku），支持从 GitHub 自动部署：

  1. Nixpacks 自动检测：发现 requirements.txt，识别为 Python 项目
  2. 构建阶段：创建虚拟环境、安装所有 Python 依赖（包括 PyTorch、sentence-transformers 等）
  3. 运行阶段：执行 uvicorn app.main:app 启动 FastAPI 服务
  4. 启动后自动 ingest：通过 lifespan 钩子在后台加载知识库数据

nixpacks.toml 的作用

类似于 vercel.json 或 netlify.toml——告诉部署平台怎么构建和启动你的应用。关键配置是强制安装 CPU 版 PyTorch（默认会装 GPU 版，3GB+，免费服务器装不下）。

[phases.install]
cmds = [
  "pip install torch --index-url https://download.pytorch.org/whl/cpu && pip install -r requirements.txt"
]

08前后端通信：CORS 和跨域

因为前端（vercel.app）和后端（belmo.uk）域名不同，浏览器会触发跨域限制。后端通过 FastAPI 的 CORS 中间件解决：

app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "http://localhost:3000",           # 本地开发
        os.getenv("FRONTEND_URL"),         # 线上前端域名
    ],
    allow_methods=["*"],
    allow_headers=["*"],
)

这跟 Next.js API Routes 里设置 Access-Control-Allow-Origin 响应头是一回事，只是 FastAPI 用中间件统一处理了。
